## 标记说明

- ✅ **绿灯**:AI 直接做,不用问,自动写日志
- 🟡 **黄灯**:AI 提交后**必须主管/老板审批**,审批通过才执行

## 留痕字段(每个动作必须记录)

写入 `action_logs` 表的字段:

```typescript
{
  actor_id: number,           // 谁干的(用户 ID)
  actor_type: 'human' | 'ai',
  shop_id: number,            // 哪个店铺
  action_type: string,        // 例 'product.price.update'
  action_payload: JSON,       // 动作参数
  before_value: JSON,         // 改前(查询类此字段为空)
  after_value: JSON,          // 改后(查询类此字段为空)
  risk_level: 'green' | 'yellow' | 'red',
  approval_status: 'auto' | 'pending' | 'approved' | 'denied',
  approved_by: number | null, // 谁审的
  ip_address: string,
  user_agent: string,
  screenshot_path: string,    // 关键动作截图路径
  created_at: timestamp,
  retention_until: timestamp  // created_at + 180 天
}
```

## 人审流程(🟡 黄灯)

```text
1. AI/员工提交任务 → 服务器写 ai_tasks 表 status=pending
2. 边界系统打标 risk=yellow,approval_required=true
3. WebSocket 通知该店铺所属团队主管:"有任务等你审"
4. 主管在客户端 UI 看到"待审清单"
5. 主管点【批准】或【驳回】
   - 批准 → status=approved → 派发到执行客户端 → 执行 → done
   - 驳回 → status=denied → 通知 AI 失败原因
6. 老板每天下班看一眼"今天黄灯流水",抽查
```

**主管不在的兜底**:
- 黄灯任务 24 小时无审批 → 自动通知老板
- 紧急任务(payload 标注 urgent=true)→ 立即通知老板,不等主管

## 红灯告警(❌)

红灯触发时(AI 试图调用永禁动作):
1. 服务器**立即拒绝**返回 403
2. 写 `emergency_events` 表 event_type='boundary_red_attempted'
3. 通过老板配置的告警通道(钉钉/微信/邮件)推送:
   ```
   ⚠️ 边界违规告警
   时间:2026-05-08 14:30
   AI:秘书 AI(id=2)
   店铺:韩国一店(id=3)
   尝试动作:account.password.change
   payload:{...}
   已拦截 + AI 冻结至 15:30
   ```
4. AI 账号设置 `frozen_until = NOW() + 1 hour`
5. AI 账号登录失败提示:"已被边界系统冻结,等待解冻或联系老板"
6. 老板手动解冻接口:`POST /admin/unfreeze/:user_id`

## 边界清单的版本管理

- 这份清单**作为代码常量**存在 `server/src/modules/boundary/rules.ts`
- 每次修改必须:
  1. 老板批准(微信/钉钉/口头都可)
  2. git commit message 以 `[boundary]` 开头
  3. 同步更新本文档(docs/02-边界系统清单.md)
- 每月由 Claude(主控)review 一次:
  - 看通过率/驳回率
  - 看哪些黄灯几乎从来不驳回(可以降级为绿)
  - 看哪些绿灯频繁出问题(应该升级为黄)
